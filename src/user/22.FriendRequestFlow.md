# 24. Friend Request Status & Schema Guide

This document defines the database schema and the detailed business semantics of each status for the Friend Request / Profile Connection feature in the matrimonial platform.

---

## 1. Database Schema (`tbl_user_friend_request`)

```sql
CREATE TABLE tbl_user_friend_request(
    id character(32) NOT NULL,
    sender_user_id character(32) NOT NULL,
    receiver_user_id character(32) NOT NULL,
    status character varying(32),
    created_by character(32),
    created_at bigint,
    updated_by character(32),
    updated_at bigint,
    deleted boolean,
    PRIMARY KEY (id),
    FOREIGN KEY (sender_user_id) REFERENCES tbl_user_details(id),
    FOREIGN KEY (receiver_user_id) REFERENCES tbl_user_details(id)
);
```

---

## 2. Status Explanations

### A. New Request (`'New Request'`)
* **Role:** Initiated by the **Sender**.
* **Meaning:** This represents a pending connection interest.
* **Platform Behavior:** 
  * The **Sender** sees it in their "Sent Interests" list as pending.
  * The **Receiver** sees it in their "Received Requests" inbox where they can choose to take action.

### B. Accepted (`'Accepted'`)
* **Role:** Triggered by the **Receiver**.
* **Meaning:** The connection request has been mutually agreed upon.
* **Platform Behavior:** 
  * Unlocks premium communication between the two users.
  * Both users can now chat and view each other's contact details (phone number, email address, etc.).

### C. Decline (`'Decline'`)
* **Role:** Triggered by the **Receiver**.
* **Meaning:** The receiver is not interested in connecting with this profile.
* **Platform Behavior:** 
  * The request is silently removed from the receiver's inbox.
  * The sender is **not** explicitly notified of the decline (maintaining positive UX), but they are strictly blocked from sending another request.

### D. Blocked (`'Blocked'`)
* **Role:** Can be triggered by **Either** the sender or the receiver.
* **Meaning:** The profile has been blocked completely.
* **Platform Behavior:** 
  * Enforces absolute invisible mode.
  * The two profiles can never see each other in New Matches, Premium, Compatibility, search results, or shortlist feeds. No future connection attempts are allowed.

### E. Ignore (`'Ignore'`)
* **Role:** Triggered by the **Receiver**.
* **Meaning:** The receiver mutes/archives the request.
* **Platform Behavior:** 
  * The request is archived and hidden from the receiver's incoming feed.
  * Crucially, the sender **still sees it as pending** (so they do not feel explicitly rejected), maintaining a polite and respectful platform environment.

---

## 3. Recommended Status Flow

* **Send Connection:** `[*] ➡️ New Request`
* **Accept Interest:** `New Request ➡️ Accepted`
* **Reject Interest:** `New Request ➡️ Decline`
* **Mute/Archive:** `New Request ➡️ Ignore`
* **Block User:** `[Any Status] ➡️ Blocked`

## 4. API Reference: Send Friend Request

Initiates a connection interest (friend request) from the authenticated user to a target user.

* **URL:** `POST` `/api/auth/user/friend-request`
* **Headers:**
  ```http
  Authorization: Bearer <user_token>
  x-app-type: user
  Content-Type: application/json
  Accept-Language: en-US
  ```
* **Request Body:**
  ```json
  {
      "target_user_id": "receiver_user_id_here"
  }
  ```
* **Successful Response (200 OK):**
  ```json
  {
      "statusCode": 200,
      "messageCode": "success.requestSent",
      "message": "Friend request sent successfully."
  }
  ```

## 9. API Reference: Update Friend Request Status

Updates the status of an existing friend request. Used by the **receiver** to Accept, Decline, or Ignore a request, or by **either** user to Block.

* **URL:** `PUT` `/api/auth/user/friend-request/status`
* **Headers:**
  ```http
  Authorization: Bearer <user_token>
  x-app-type: user
  Content-Type: application/json
  Accept-Language: en-US
  ```
* **Request Body:**
  ```json
  {
      "sender_user_id": "sender_id_hex",
      "status": "Accepted"
  }
  ```

  > **Allowed status values:** `Accepted` | `Decline` | `Ignore` | `Blocked`

  > **Note:** The backend identifies the request record using `sender_user_id` (from request body) + the logged-in user's ID (from auth token as receiver). No separate `request_id` is needed.

* **Successful Response (200 OK):**
  ```json
  {
      "statusCode": 200,
      "messageCode": "success.statusUpdated",
      "message": "Friend request status updated successfully."
  }
  ```

## 11. API Reference: Get Friend Request Counts

Retrieves a summary count of all friend request categories for the authenticated user in a single call.

* **URL:** `GET` `/api/auth/user/friend-request/counts`
* **Headers:**
  ```http
  Authorization: Bearer <user_token>
  x-app-type: user
  Accept-Language: en-US
  ```
* **Successful Response (200 OK):**
  ```json
  {
      "newRequestCount": 5,
      "ignoredCount": 2,
      "acceptedCount": 10,
      "sentCount": 8,
      "receivedCount": 15,
      "totalCount": 23
  }
  ```

### Field Descriptions

| Field | Description |
| :--- | :--- |
| `newRequestCount` | Requests received by me that are still **pending** (`status = 'New Request'`, `receiver = me`) |
| `ignoredCount` | Requests received by me that I have **ignored** (`status = 'Ignore'`, `receiver = me`) |
| `acceptedCount` | Requests received by me that I have **accepted** (`status = 'Accepted'`, `receiver = me`) |
| `sentCount` | Requests **sent by me** that are still pending (`status = 'New Request'`, `sender = me`) |
| `receivedCount` | Total requests ever **received by me** (all statuses, `receiver = me`) |
| `totalCount` | Grand total = all requests **sent by me** (all statuses) + all requests **received by me** (all statuses) |
